# Cosmos3 Closed-Loop Eval Code Audit

Date: 2026-06-13 CST

## Trigger

The user stopped further SFT because more training is likely overfitting and
asked to re-read the idea/plan and inspect whether the closed-loop eval code is
wrong.

## What The Plan Requires

`IDEA.md` and `PLAN/cosmos3_300f_world_model/03_testing_and_metrics.md` define
the controller-facing loop as:

- observe live RGB/state;
- build a causal Cosmos prefix from the latest observed frames/state;
- predict a remaining or short future task rollout;
- execute only a short action prefix;
- reobserve the real simulator state;
- refresh the prefix and repeat;
- hand back to frozen DP only when the real relative state is continuable.

The frozen DP is a static skill prior. It should not be asked to solve the
absolute dynamic OOD segment by a long blind takeover.

## What The Current Code Actually Does

`scripts/world_model/run_cosmos3_receding_closed_loop.py` currently implements
a guarded diagnostic smoke:

- recover an eval sample and source H5;
- restore the simulator to `env_states[chunk_start]`;
- execute one precomputed Cosmos robot-action chunk, usually 8 steps;
- optionally run frozen DP for `dp_resume_horizon` steps from live observations;
- record live simulator metrics and video.

It does not rerun Cosmos after live re-observation. It does not implement a
continuability classifier or guard before long DP resume. It does not feed a
Cosmos-predicted future task-state/reconstruction interface to DP. It also
does not continue the source H5 external target motion after the prefix reset:
moving-hole fix3 rows were generated by manually setting the target box pose
after each action step, not by installing an autonomous target-motion
controller in the environment. Therefore the existing `8+96` DP panels are
only one-shot-chunk plus long-DP-takeover diagnostics in an incomplete dynamic
world.

## Code-Level Checks

- Action alignment looks structurally consistent in inspected `iter2100`
  samples: `prefix_frame_index`, `future_start_action_index`, and
  `chunk_start` match, e.g. `f097 -> action[97:105]`.
- The DP resume bug that previously executed only one 8-step DP block had
  already been fixed; the current loop recomputes DP actions from the latest
  live observation for repeated DP chunks.
- A more serious physical-eval bug was confirmed after inspecting the fix3
  generator and source H5 sample. The generator applies moving-hole dynamics
  with `base_env.box.set_pose(...)` after `env.step(action)`. The inspected
  `hole_late_move_stop_seed3280649_idx2518` source is static through frame
  `104`, starts target motion at frame `105`, reaches final target pose around
  frame `128`, and moves about `0.224m`. A live eval restored at frame `97`
  must keep applying that external target pose schedule while executing live
  robot actions. Otherwise it is no longer evaluating the planned dynamic
  task.
- The bigger problem is protocol mismatch, not extra SFT steps. The live eval
  asks whether one Cosmos chunk puts the simulator into a state where frozen
  DP can run for a long horizon, not whether the planned receding world-model
  handoff works.

## Fix Applied

- Added `--allow-long-dp-diagnostic` to
  `scripts/world_model/run_cosmos3_receding_closed_loop.py`.
- In `MODE=smoke`, any `dp_resume_horizon > action_exec_horizon` now fails
  unless that diagnostic override is explicit.
- Changed `scripts/slurm/run_cosmos3_closed_loop_panel_in_allocation.sh` so
  panel runs no longer default to `VISUAL_REVIEW_STATUS=pass` or DP96.
- Added manifest/summary fields:
  `controller_interface=one_shot_cosmos_chunk_then_optional_frozen_dp_resume`
  and `method_evidence_allowed=false`.
- Added the first per-reobservation Cosmos inference primitive:
  `scripts/world_model/build_cosmos3_live_prefix_wam_input.py` and
  `scripts/slurm/run_cosmos3_live_prefix_policy_inference_in_allocation.sh`.
  This builds one causal policy-mode JSONL row from an observed prefix video
  and history action/state rows, then can invoke Cosmos inference inside a
  compute-node allocation. It is not a full controller loop by itself.
- Added `scripts/world_model/extract_cosmos3_policy_action_chunk.py` and
  connected it to the live-prefix wrapper so a real inference output can be
  converted into a short de-normalized robot-action chunk without using
  sidecar columns as simulator state.
- Tightened the RGB causal boundary: the live-prefix builder now rejects a
  video whose frame count is not exactly `prefix_frame_index + 1`, unless an
  explicit diagnostic override is passed. `scripts/world_model/write_video_prefix.py`
  writes prefix-only videos from longer references.
- Added `scripts/world_model/run_cosmos3_live_receding_loop.py`, a
  compute-node-only scaffold that restores a source prefix, writes the
  prefix-only observed video and raw `300x32` WAM history, and can optionally
  run Cosmos/extract/execute/reobserve iterations.
- Repaired the live receding loop's dynamic-world semantics. It now defaults
  to `--external-target-mode source_env_state`: after each live robot action,
  it replays only `env_states/actors/box_with_hole[frame+1]` from the source
  H5, then reads real task state, appends the live WAM history row, and
  renders the new observation. It does not restore source robot or peg state
  after the prefix.
- Repaired the DP handoff observation path in the live receding loop. Because
  `env.step()` returns an observation before the external target replay, the
  loop now maintains its own latest two post-replay `base_env.get_obs()` state
  observations and feeds those to frozen DP if the continuability gate passes.

## Prefix-Start Correction

The fixed-prefix live runs that used manifest frame `97` on sample
`hole_late_move_stop_seed3280649_idx2518` were a diagnostic mistake for the
active method. `--prefix-role auto` only changed the role after a manually
chosen initial prefix; it did not decide when the world model should engage.
That does not satisfy the planned controller interface, where the system keeps
observing under the base policy and switches the world-model path when target
motion is detected from the observation stream.

The live loop now separates these two choices:

- `--prefix-start-mode target_motion_onset`: active default for live-receding
  panel runs. It scans observed `box_with_hole` poses causally, checks target
  delta/speed thresholds, and triggers only after the configured consecutive
  moving-frame rule is satisfied.
- `--prefix-start-mode manual`: explicit diagnostic/backward-compatible
  override. It records `method_evidence_allowed=false`.

On the same source sample, the dynamic detector selected frame `106`: frame
`105` was the first observed moving frame and frame `106` was where the
two-frame consecutive rule became observable. This replaces the invalid
hand-picked manifest frame `97` for corrected live-receding evaluation.

The partial fixed-prefix `long8` and `long7` live eval steps on allocation
`127350` were stopped at the process/step level without cancelling the held
allocation. Their partial artifacts must not be used as method evidence.

One more fixed-prefix entry point was found after the dynamic-trigger smoke:
`scripts/slurm/run_cosmos3_live_receding_loop_in_allocation.sh` still defaulted
to `PREFIX_FRAME_INDEX=97`. A stray `long7_v3` step launched through that
wrapper on `127350.35`; it was stopped without cancelling the held allocation.
The wrapper now defaults to `PREFIX_START_MODE=target_motion_onset`,
`PREFIX_FRAME_INDEX=-1`, and the same consecutive-frame target-motion trigger
used by the panel wrapper. Its partial `iter_00_prefix_f097` artifacts are
invalid diagnostics.

## DP Handoff State-Machine Correction

After re-reading the plan again for the user's 2026-06-13 concern, one more
protocol mismatch remained in `run_cosmos3_live_receding_loop.py`. The live
loop had stopped using the old one-shot DP96 wrapper, but once a real-state
`C_pi` gate passed it could still execute the full `dp_handoff_horizon`
(`32`/`64` steps by wrapper default) and then stop the sample. That is still
too close to blind DP takeover.

The corrected controller state machine is now:

- build the latest observed RGB/action/state prefix;
- compute `C_pi` from the real live simulator state before choosing a
  controller;
- if `C_pi` passes, execute only a short frozen-DP chunk bounded by
  `min(action_exec_horizon, dp_handoff_horizon)`;
- after that DP chunk, reobserve and return to the same decide loop unless
  the real simulator succeeded, terminated, or reached the 300-step boundary;
- if `C_pi` fails, run one short Cosmos rebind chunk, then reobserve and loop.

The new records distinguish `controller_step_type=frozen_dp_short_chunk` from
`controller_step_type=cosmos_rebind_short_chunk`, and record both
`pre_controller_continuability_gate` and `post_cosmos_continuability_gate`.
This keeps DP as the static skill prior and prevents closed-loop evidence from
silently becoming "one Cosmos chunk plus long DP takeover."

Only local static checks were run for this fix:

- `py_compile` on the live receding loop, panel, live-prefix builder, and
  action extractor;
- `bash -n` on the live-receding loop/panel/prefix wrappers;
- `git diff --check`.

No SFT training, Cosmos inference, or simulator rollout was launched for this
audit fix.
