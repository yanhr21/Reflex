# Dataset Construction Record

Date: 2026-07-06

This document records the current method setting and dataset design so it is
not lost in later implementation work.

## Method Setting

The base policy is a state-based Diffusion Policy trained for static
`PegInsertionSide-v1`. It can output successful static insertion actions, but
it is not expected to solve dynamic target / moving hole scenes by itself.

Cosmos-3 should remain RGB-based. Its purpose is to observe dynamic visual
history and imagine future target state. The output needed by the controller is
not raw Cosmos action. The needed output is:

```text
future RGB / latent
-> future target or hole frame
-> timing tau
-> target velocity
-> uncertainty
```

The controller-facing method is:

```text
real current robot state
+ real current peg / gripper state
+ Cosmos-predicted future hole frame
+ moving-frame adapter
+ protected state-based DP skill
-> legal DP-compatible action
```

The key idea is that the DP performs static insertion in a predicted future
target frame. For a continuously moving target, the system must predict where
the target will be at contact / insertion time, not merely where it is now.

## Why Final Position Alone Is Not Enough

A single future position can be enough only for slow or nearly stationary
targets. For persistent motion, insertion depends on:

- future target / hole pose;
- target velocity at contact;
- estimated time-to-contact `tau`;
- relative peg-hole velocity;
- whether the robot can reach the future frame within its velocity and
  acceleration limits;
- uncertainty in the prediction.

Therefore the dataset must supervise a future target trajectory window, or at
least future pose + velocity + tau + uncertainty.

## Current Asset Status

Official DP:

- trained on state/action data, not RGB;
- uses `pd_ee_delta_pose`;
- official checkpoint should be protected and not fully fine-tuned at the
  start.

Official 1000 demos:

- available as H5/JSON state/action data;
- useful as static expert data;
- need RGB rendering before Cosmos can use them for visual training.

Old `fix3_733`:

- useful as reference, context, and maybe early Cosmos/readout material;
- not enough as final dataset because many samples were generated by or gave
  opportunity to the original DP checkpoint in moving scenes;
- should not be treated as proof that the new method solves a new dynamic
  setting.

## Data Split By Training Role

### A. Static Expert Data

Use official 1000 demos.

Training role:

- DP action BC / distillation;
- static insertion phase extraction;
- Cosmos static future prediction after RGB rendering;
- static regression check.

### B. Dynamic RGB Observation Data

Generate moving target / moving hole / peg disturbance episodes. Success is
not required.

Training role:

- Cosmos future RGB / latent prediction;
- target trajectory and velocity readout;
- uncertainty and future-frame consistency.

Do not train DP to imitate failed action chunks from this data.

### C. Frozen-DP Dynamic Outcome Data

Run the frozen official DP in dynamic scenes.

Training role:

- outcome labels for success, miss, jam, no-progress, and target-assisted
  cases;
- discrepancy / OOD signal;
- infeasible, miss, jam, target-assisted, no-progress classification;
- contrastive separation between insertion-corridor futures and failure
  futures.

Success is allowed and must be recorded as an outcome label. C is invalid only
when the outcome depends on state intervention, snap / teleport, target-assisted
self-insertion, or another forbidden hidden intervention. Do not force frozen-DP
success into failure or archive it merely because it succeeded.

### D. Future-Frame Cooperation Teacher Data

Generate legal rollouts where a teacher uses ground-truth future target
trajectory to execute moving-frame insertion actions.

Training role:

- moving-frame adapter / residual supervision;
- phase and timing supervision;
- relative velocity at contact;
- active robot insertion under moving target.

The teacher may use ground-truth future only for data generation. It must be
recorded as teacher-only and cannot be claimed as deployed method success.

### E. Cosmos-Predicted Cooperation Data

Replace ground-truth future target frame with Cosmos/readout prediction.

Training role:

- close the train-test gap;
- adapter robustness to Cosmos prediction error;
- uncertainty-conditioned live control.

This comes after A-D. It should not be the first dataset.

## Loss Permissions

Use DP BC loss only on true expert or validated cooperation-teacher actions.

Use Cosmos future loss on RGB/state-aligned static and dynamic data.

Use negative/discrepancy/contrastive losses on failed DP dynamic data and
target-assisted cases.

Use adapter/residual loss only on data where the intended future-frame
condition and legal action trace are known.

Do not silently mix all samples into one generic success dataset.

## Acceptance And Rejection

Positive method or adapter data requires:

- active robot-driven insertion;
- continuous target motion if the target is moving;
- legal controller actions;
- aligned RGB and state/action timing;
- no state edit, snap, target-assisted self-insertion, or hidden manual
  finisher.

Negative/diagnostic data includes:

- DP misses;
- target-assisted insertion;
- jam or bad relative velocity;
- plausible Cosmos future that fails executability.

Reject from active training if:

- the outcome depends on `set_pose`, `set_state`, source restore, saved-state
  replay, snap, or geometric final placement;
- timestamps cannot be aligned;
- labels are ambiguous and cannot be audited.

## First Experimental Gate

Before training Cosmos into the loop, test the interface with ground-truth
future target frame:

```text
current real state
+ GT future hole pose / velocity / tau
+ protected DP + adapter
-> moving-target insertion attempt
```

If this fails, Cosmos prediction quality is not the primary blocker; the
moving-frame representation or adapter is wrong. If this succeeds, train
Cosmos/readout to predict that future target frame from RGB history.
